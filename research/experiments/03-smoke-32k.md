# 实验 03-smoke：3 臂 × 32K 窗口（首次烟测）

**日期**：2026-06-25
**配置**：8 dossier ~32K 语料，5 问，hard-window 32K，docs 5K tokens each。

## 结果

| 臂 | terminated_early | accuracy | peak view tokens | total prompt | 治理 |
|---|---|---|---|---|---|
| react | false | 100% | **9,647** | 31,917 | 0/0/0 |
| threshold | false | 100% | **2,839** | 18,231 | 0/0/0 |
| cpat | false | 100% | **5,231** | 55,154 | 0/0/0 |

## 诊断（关键，第 6 次踩同一个坑）

**全部 100%、peak tokens 远低于窗口、零治理。** 没有任何一臂溢出，react 也没提前终止。

**根因**：任务**可被 grep 一击解决**。每个事实就在一次 `grep_search "project Aurora"` 之外——
agent 搜到、读 3 行、作答，**从不需要把语料留在 context 里**。于是受限窗口根本不咬人，CPAT 无事可做。

> 这是整个研究反复出现的**同一个失败模式**：只要任务允许"选择性检索 + 即时作答"，
> 上下文就不会累积，治理就无价值。CAT 论文用 SWE 任务（必须反复执行命令、读大量
> observation、跨多步推理）天然强制累积；我用"查事实"任务则天然可被 grep 绕过。

## 修复方向（写入设计 v2）

要让累积**不可避免**，任务必须满足以下之一：
1. **答案需要综合一整篇文档**（不是单行事实）——强制 read_file 全文而非 grep 片段。
2. **答案需要跨多篇文档聚合**（如"列出所有 accuracy > 80% 的项目"——必须读遍全部）。
3. **禁用 grep_search**，只给 list_dir + read_file（强制整篇读）。
4. **问题不含可 grep 的关键词**（语义匹配而非字面）。

最强的是 **2 + 3 组合**：聚合类问题（必须读遍语料）+ 只能整篇读（无 grep 跳过）。
这样 react 必然把多篇大文档堆进 context → 溢出 → 提前终止；CPAT 可边读边治理。

## 价值

烟测**成功拦截**了一次昂贵的无效全量实验。这正是先跑 smoke 的意义。
