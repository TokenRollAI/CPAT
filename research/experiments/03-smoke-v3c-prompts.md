# 实验 03-smoke-v3c：新提示后的 cpat（提示工程的极限）

**日期**：2026-06-25
**配置**：32K 窗口，20 docs ~100K，埋深事实回访（同 v3/v3b），cpat + 三层新提示 + 护栏。

## v3 系列完整对照

| 臂/版本 | accuracy | revisit | total prompt | compact | offload | turns | 备注 |
|---|---|---|---|---|---|---|---|
| react | 10% | 0% | 58,755 | 0 | 0 | 60 | 提前终止 |
| threshold | 100% | 100% | **142,474** | 0 | 18 | 12 | 被动基线 |
| cpat v3（旧提示） | 90% | 50% | 445,786 | 9 | 14 | 29 | 语义漂移 |

## 新提示的真实成效（事实）

**正面**：
1. **compact 9→0**：新提示成功把 agent 从有损 compact 导向安全可逆的 offload（22 次）。
2. **accuracy 稳 100%**，peak view tokens 仅 20,240——里程碑 offload 让 context 始终稳压窗口内。
3. agent 学会复用前序扫描（Q1/Q2 答"I already have all the data from Q1"），没重读。

**负面**：
1. **total token 445K，比 v3b(319K) 更高，远超 threshold(142K)**：新提示让 agent 更勤治理，
   但**每次 context_update 都是一次额外 LLM 往返**，开销累积 > 节省。
2. **Q8/Q9 仍出现 "No new question visible" 漂移苗头**（虽答对）：护栏防住了答错，没根除困惑。

## 决定性结论（v3 三轮后，有把握）

> **在"信息源可重读"的任务上，prompt-only 的主动 CPAT 无法在效率上赢过被动 threshold。**
> 怎么调提示都一样：**主动治理的 LLM 往返开销 > 它节省的**。

这**精确印证 CAT 论文核心论点**：CAT 赢，是因为用 **SFT 训练**让模型在推理过程中
**零额外往返地**顺带做治理决策；而我们的 CPAT 把"治理作为独立 tool call"，
**每次治理都有固定往返开销，prompt 再好也省不掉**。

**瓶颈不在提示，在架构**——这比"提示没调好"更深、更有价值的发现。

## 三个维度的最终判定

| 维度 | 判定 | 证据 |
|---|---|---|
| 治理 > 不治理（受限窗口） | ✅ **成立** | react 10% → 治理臂 100% |
| 主动 CPAT > 被动 threshold（质量） | ⚠️ **打平** | 都 100%（加护栏后） |
| 主动 CPAT > 被动 threshold（效率） | ❌ **不成立** | cpat 445K vs threshold 142K token |
| restore 的独立价值 | ❓ **未触发** | 文件可重读 → restore 始终 0 |

## 仍未关闭的问题（restore 的价值）

所有"主动 > 被动"的失败都有一个共因：**信息源（文件）可重读**，所以被动 offload + 重读永远够用，
restore 永远用不上。要让主动 CPAT（尤其 restore）真正体现独立价值，**必须换到信息源不可重得的任务**
——这是唯一还没试、且理论上 CPAT 该赢的场景。
